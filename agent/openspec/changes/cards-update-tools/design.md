## Context

Currently, the process of updating new cards for LumiTracker is manual. The source cards data are placed in the `./cards` directory, and source CSVs are in `./cards/generated` (originally generated by a human-maintained Excel table, `./【七圣】多语言文本&卡牌数据.xlsx`). The Excel table cannot be modified programmatically right now, so the update process involves adding new data to CSVs, and manually updating the Excel later.

The new data is typically found on `https://gi.yatta.moe`. We need python tools to automate fetching the data and integrating it into the `./cards` database to reduce manual effort. The data includes text info in three languages (chs, en, jp) and high-quality, uncompressed card images (420*720 resolution).
Exploration shows that the changelog is an SPA, which can be problematic to scrape (especially on WSL1 where node execution can fail). However, we can reliably discover new cards by fetching the bulk JSON API and diffing it against the local database CSVs by name.

## Goals / Non-Goals

**Goals:**
- Automate fetching new card data from `https://gi.yatta.moe/api/v2/{lang}/gcg` by finding cards that do not yet exist in the local `cards/generated` CSVs.
- Download uncompressed PNG images (420x720) for the new cards.
- Generate a **single** intermediate CSV file (`temp_cards.csv`) containing a union of all required columns.
- Automatically populate the website's ID into the `id` column, but leave `share_id` blank for manual filling.
- Automate updating the `./cards/generated/*.csv` database and moving images into the correct subdirectories using the manually corrected intermediate CSV.
- Add a validation step to test the pipeline using v6.4.

**Non-Goals:**
- Automate the update of the human-maintained Excel table (`./【七圣】多语言文本&卡牌数据.xlsx`). This will still be done manually later.
- Automatically infer the correct `share_id` from the website. Manual intervention is explicitly required.

## Decisions

- **Two-Step Process**: The update process is split into two scripts (Task 1: Fetch, Task 2: Update). 
  - *Rationale*: A human needs to intervene to fill the `share_id` and review translations in a single unified CSV.
- **API Diffing Approach**: Fetch the entire bulk JSON API (`/api/v2/{lang}/gcg`) and diff the `name` (zh-HANS) against the local `actions.csv`, `characters.csv`, and `tokens.csv`. Any name not found locally is considered a new card.
  - *Rationale*: Playwright is unnecessary, avoids WSL1 environment execution errors, and is significantly faster and more robust.
- **Single Temporary CSV**: Downloaded data for a specific version will be stored in `./agent/temp/updates/{version}/temp_cards.csv`.
- **Target Image Destinations**: The update script will distribute images based on their resolved type and ID directly to `cards/characters/`, `cards/actions/`, or `cards/actions/tokens/` as parsed from `watcher/database.py`.

## Risks / Trade-offs

- **[Risk] Name changes** → If the local name in the database differs slightly from the Yatta API name, it will falsely identify an existing card as new. *Mitigation*: The user will catch these duplicates during the manual review step in the unified CSV.
