## Context

Currently, the process of updating new cards for LumiTracker is manual. The source cards data are placed in the `./cards` directory, and source CSVs are in `./cards/generated` (originally generated by a human-maintained Excel table, `./【七圣】多语言文本&卡牌数据.xlsx`). The Excel table cannot be modified programmatically right now, so the update process involves adding new data to CSVs, and manually updating the Excel later.

The new data is typically found on `https://gi.yatta.moe`. We need python tools to automate fetching the data and integrating it into the `./cards` database to reduce manual effort. The data includes text info in three languages (chs, en, jp) and high-quality, uncompressed card images (420*720 resolution).
Exploration shows that the changelog is an SPA, which can be problematic to scrape (especially on WSL1 where node execution can fail). However, we can reliably discover new cards by fetching the bulk JSON API and diffing it against the local database CSVs by name.

## Goals / Non-Goals

**Goals:**
- Automate fetching new card data from `https://gi.yatta.moe/api/v2/{lang}/gcg` by finding cards that do not yet exist in the local `cards/generated` CSVs.
- Download uncompressed PNG images (420x720) for the new cards. Avatar images (256x256) cannot be fetched reliably, so define a file naming pattern and instruct the user to download them manually.
- Generate 3 separate intermediate CSV files (`actions.csv`, `characters.csv`, `tokens.csv`) matching the local `./cards/generated` headers exactly. Generate a `TODO-list.md` outlining manual steps, and save raw JSON data.
- Automate updating the `./cards/generated/*.csv` database (including `share_code.csv`) and moving images into the correct subdirectories using local sequential `internal_id` for filenames. Validate CSVs before updating.
- Add a validation step to test the pipeline and detailed test cases for edge scenarios (missing data, artifacts, empty token files).

**Non-Goals:**
- Automate the update of the human-maintained Excel table (`./【七圣】多语言文本&卡牌数据.xlsx`). This will still be done manually later.
- Automatically infer the correct `share_id` from the website. Manual intervention is explicitly required.

## Decisions

- **Two-Step Process**: The update process is split into two scripts (Task 1: Fetch, Task 2: Update). 
  - *Rationale*: A human needs to intervene to fill the `share_id`, review translations, and manually download avatar images.
- **API Diffing Approach**: Fetch the entire bulk JSON API (`/api/v2/{lang}/gcg`) and diff the `name` (zh-HANS) against the local `actions.csv`, `characters.csv`, and `tokens.csv`. Any name not found locally is considered a new card.
  - *Rationale*: The changelog is an SPA without a clean version-specific API endpoint. Diffing is significantly faster and more robust than scraping.
- **Three Temporary CSVs**: Downloaded data will be split directly into `./agent/temp/updates/{version}/` as `actions.csv`, `characters.csv`, and `tokens.csv` matching the local formats. A `TODO-list.md` will guide the user.
- **Target Image Destinations**: The update script will assign sequential local internal IDs, and distribute images directly to `cards/characters/`, `cards/actions/`, or `cards/actions/tokens/`. Avatar images provided by the user (as `avatar_{icon_name}.png`) will be migrated to `cards/avatars/`.

## Risks / Trade-offs

- **[Risk] Name changes** → If the local name in the database differs slightly from the Yatta API name, it will falsely identify an existing card as new. *Mitigation*: The user will catch these duplicates during the manual review step in the unified CSV.

## Usage

Run these commands from the project root directory. If you encounter encoding issues in the terminal, set `PYTHONIOENCODING=utf-8`.

### Step 1: Fetch New Data
Fetch data from the Yatta API and identify new cards by diffing against the local database.
```bash
python.exe agent/scripts/fetch_cards_data.py <version>
# Example: python.exe agent/scripts/fetch_cards_data.py v6.4
```
This generates files in `./agent/temp/updates/<version>/`. Follow the instructions in the generated `TODO-list.md`.

### Step 2: Update Database
After manually filling the `share_id` and other missing data in the temp CSVs, and placing character avatar images (as `avatar_{icon_name}.png`) in the temp `images/` folder, run the update script.
```bash
python.exe agent/scripts/update_cards_db.py <version>
# Example: python.exe agent/scripts/update_cards_db.py v6.4
```
This will assign sequential internal IDs, update the main CSV databases, append to `share_code.csv`, and migrate all images to their final locations.
